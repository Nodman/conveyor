# Council — multi-model design deliberation

## What

`/conveyor:council` — a deeper, opt-in alternative to plain brainstorming for
hard design questions. Agents from different model families independently
research and propose, cross-examine each other once, then the main session
(Fable) merges the strongest parts into one verdict that feeds the normal
spec flow. Plain brainstorming is untouched.

## Why

- Cross-model scrutiny catches weak points a single model family misses.
- Cost efficiency: codex (`gpt-5.6-sol`) is far cheaper per token and proved
  capable in smoke tests (research task ~7k tokens, rebuttal ~13k).

## Decisions (locked)

- Debate shape: **one rebuttal round** — independent proposals → each attacks
  or concedes the others point by point → Fable judges and merges. No
  convergence loops.
- Invocation: **own skill** `/conveyor:council`. Hands off to the existing
  brainstorming tail (spec + user gate) after the verdict.
- Membership: **config-driven** — `council.members` in `.claude/conveyor.json`.
  Default pair: claude `fable-5` + codex `gpt-5.6-sol`.
- Agent naming: always `<runner>-<model>` (e.g. `codex-gpt-5.6-sol`,
  `claude-fable-5`) for tracing. Applies to any future external runner too.
- Codex sandbox: always `-s read-only`. The council thinks; it never edits.
- Visibility fallback is a stored user preference (`externalAgents.*`),
  reusable by future codex worker agents; asked once, settable via
  init/doctor.

## Design

### Config (`.claude/conveyor.json`)

```json
"council": {
  "members": [
    { "runner": "claude", "model": "fable-5" },
    { "runner": "codex",  "model": "gpt-5.6-sol" }
  ]
},
"externalAgents": {
  "fallbackVisibility": "window" | "background"
}
```

- `runner: "claude"` → Agent tool, `model` passed through, named agent.
- `runner: "codex"` → `codex exec` via helper script, `-m <model>`.
- `externalAgents.fallbackVisibility` unset → skill asks once, writes it.

### Helper script `plugin/scripts/codex-exec.sh`

Spawns one codex run; prints report path + session id. Responsibilities:

- Build command: `codex exec -s read-only -m <model> -o <scratch>/<name>-r<N>.md <prompt>`
  (round 2: `codex exec resume <session-id> …` — explicit id, never `--last`,
  concurrent-safe).
- Visibility ladder:
  1. `$TMUX` set → `tmux split-window` (pane shows live output, closes after
     a short delay).
  2. iTerm detected (`LC_TERMINAL` / `TERM_PROGRAM` = `iTerm.app`) → iTerm
     split pane via osascript.
  3. Else `fallbackVisibility=window` (macOS) → new Terminal window via
     osascript.
  4. Else → plain foreground/background execution, log file announced.
- Completion signal: sentinel file `<report>.done` touched after codex exits
  (works in every mode; the skill polls/waits on it).
- Session id: parsed from `codex exec` header output, echoed for round 2.

Smoke-verified primitives (2026-07-11, codex-cli 0.144.1): `codex exec -s
read-only -o <file>` headless run, tmux pane spawn + sentinel, `codex exec
resume` session continuity, Terminal-window spawn via osascript.

### Skill flow (`plugin/skills/council/SKILL.md`)

1. **Frame.** Short Q&A to pin the design question (brainstorming style,
   scaled down). Preflight: `codex` on PATH + authed; missing → report to
   user, offer plain brainstorm.
2. **Round 1 — proposals.** All members get the identical brief (question,
   repo pointers, condensed report format) and run in parallel. Claude
   members via Agent tool; codex members via helper script.
3. **Round 2 — rebuttal.** Each member receives the other proposals: "attack
   or concede point by point, then output your revised proposal." Claude
   members continue via SendMessage; codex members via `codex exec resume`.
4. **Verdict.** Main Fable judges: agreements = settled; disagreements
   resolved with stated reasoning; merged design presented to user.
5. **Hand-off.** On approval, verdict flows into the standard spec via the
   brainstorming tail. Spec records the condensed verdict and who argued
   what; raw transcripts stay in scratch.

### Error handling

- Member fails or times out → council proceeds with remaining members and
  says so; a one-member council degrades to plain deep brainstorm with a
  warning.
- codex missing/unauthed → caught in preflight, never mid-run.
- No tmux/iTerm, preference unset → ask, write config, continue.

### Testing

- Bats (`running-tests` gate): `codex-exec.sh` — visibility ladder detection,
  arg building (model, resume id, output paths), sentinel creation, name
  derivation. codex itself mocked.
- Skill flow: manual QA per `running-the-app`.

### Doctor/init touch

- `init`: asks the visibility question when scaffolding, writes
  `externalAgents.fallbackVisibility`.
- `doctor`: flags missing `council`/`externalAgents` keys when the skill is
  present; proposes defaults.

## Out of scope (v1)

- Councils inside `/conveyor:auto` runs.
- More than one rebuttal round.
- codex as an implementation (write-mode) worker agent — next feature; this
  spec's config/visibility groundwork is reused there.
- Non-macOS separate-window spawn.
