# Conveyor v1 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build the conveyor Claude Code plugin — board scripts, hooks, generic agents, lean process skills, init/doctor/work — tested with bats, through milestone 5 of the spec (live smoke green).

**Architecture:** A Claude Code plugin repo. `scripts/` are plain bash + `gh` + `jq`, parameterized entirely by the target repo's `.claude/conveyor.json`; skills carry process and call scripts via `${CLAUDE_PLUGIN_ROOT}`; two generic agents; two hooks. Tests: bats-core with a stubbed `gh` replaying fixtures.

**Tech Stack:** bash, gh CLI, jq, bats-core, GitHub GraphQL (ProjectsV2), Claude Code plugin manifest.

**Spec:** `docs/specs/2026-07-10-conveyor-design.md`. This plan covers milestones 1–5. Milestones 6–7 (cooqa-swift dogfood + self-host) get their own plan after v1.

## Global Constraints

- Runtime deps: bash, `gh`, `jq` only. Dev deps: bats-core, shellcheck.
- Config file is `.claude/conveyor.json` (spec amended from `.yml` — jq needs no extra dep).
- Board ids/names live ONLY in conveyor.json. No id or owner literal in any script/skill/agent.
- All scripts: `set -euo pipefail`, shellcheck-clean, source `scripts/lib.sh`.
- Skills are user-invocable (`/conveyor:<name>`); NO `commands/` directory.
- Canonical status keys everywhere: `humanOnly backlog ready inProgress agentReview qa done archived`. Priority keys: `p1 p2 p3`. Labels: `approved-by-agent`, `qa-passed`.
- Docs/comments/skills: condensed style — short sentences, bullets, no filler. Comments only for constraints code can't show.
- Every commit message: conventional prefix (`feat:`/`test:`/`docs:`/`chore:`), imperative, one line unless a body is needed.

```bash
#!/usr/bin/env bash
set -euo pipefail
source "$(dirname "${BASH_SOURCE[0]}")/lib.sh"
need gh; need jq

if [[ "${1:-}" == "--find" ]]; then
  owner="$2"; repo="$3"
  # first project whose title or linked repositories match; fall back to title==repo
  num=$(gh project list --owner "$owner" --format json \
    | jq -r --arg r "$repo" '[.projects[] | select(.title==$r)][0].number // empty')
  [[ -n "$num" ]] || exit 3
  echo "$num"; exit 0
fi

owner="$1"; number="$2"
fields=$(gh project field-list "$number" --owner "$owner" --format json)
canon_status='{"humanOnly":"Human Only","backlog":"Backlog","ready":"Ready for dev","inProgress":"In Progress","agentReview":"Agent Review","qa":"QA","done":"Done","archived":"Archived"}'
canon_prio='{"p1":"P1","p2":"P2","p3":"P3"}'

jq -n --argjson f "$fields" --argjson cs "$canon_status" --argjson cp "$canon_prio" '
  def opts(fname): ([$f.fields[] | select(.name==fname)][0]) as $fld
    | {id: ($fld.id // null),
       map: (reduce ($fld.options // [])[] as $o ({}; .[$o.name] = $o.id))};
  (opts("Status")) as $s | (opts("Priority")) as $p |
  { projectId: ($f.fields[0].id | if . then (input_filename // null) else null end),
    statusFieldId: $s.id,
    status: ($cs | with_entries(.value as $n | .value =
      (if $s.map[$n] then {name:$n, id:$s.map[$n]} else null end))),
    priorityFieldId: $p.id,
    priority: (if $p.id then ($cp | with_entries(.value as $n | .value =
      (if $p.map[$n] then {name:$n, id:$p.map[$n]} else null end))) else null end) }'
```

Note: `projectId` is not in field-list output — fetch it with `gh project view "$number" --owner "$owner" --format json | jq -r .id` and inject via `--arg pid`; add fixture `project_view.out` (`{"id":"PVT_kwTEST","number":7}`) and simplify the jq accordingly (the snippet above shows the mapping approach; implementer wires `projectId: $pid`).

```bash
#!/usr/bin/env bash
set -euo pipefail
source "$(dirname "${BASH_SOURCE[0]}")/lib.sh"
need gh; need jq
cmd="$1"; issue="$2"

item_row() {
  gh project item-list "$(cfg .project)" --owner "$(cfg .owner)" --limit 200 --format json \
    | jq -r --argjson n "$issue" \
      '.items[] | select(.content.number==$n) | "\(.id)\t\(.status // "")"' | head -1
}

row=$(item_row); [[ -n "$row" ]] || die_code3 "no card for issue #$issue"  # exit 3
case "$cmd" in
  find) printf '%s\n' "$row" ;;
  move)
    key="$3"
    gh project item-edit --project-id "$(cfg .projectId)" \
      --field-id "$(cfg .statusFieldId)" \
      --single-select-option-id "$(status_id "$key")" \
      --id "${row%%$'\t'*}" >/dev/null
    echo "moved #$issue → $(status_name "$key")" ;;
  *) die "usage: card.sh find|move ISSUE [STATUS_KEY]" ;;
esac
```

(Add `die_code3` to lib.sh: like `die` but exit 3; test it in lib.bats with one extra case.)

description: Use when implementing any feature or bugfix that has a testable seam, before writing implementation code. Red → green → refactor.

# TDD

The loop, per behavior:
1. **Red** — write ONE failing test for the next behavior. Run it. It must
   fail for the RIGHT reason (missing behavior, not a typo/import error).
2. **Green** — write the minimum code that passes. Run the test. Passes.
3. **Refactor** — clean up with tests green. Run the full touched suite.
4. Commit. Next behavior.

- Test the behavior, not the implementation. One behavior per test; exact

- Never weaken, skip, or delete a failing test to get to green — fix the code,

  own step.
- Bugfix = first a test that reproduces the bug, then the fix.
- The failing run in step 1 is not optional — a test you never saw fail proves
  nothing.

files, throwaway scripts): say so explicitly in one line ("TDD n/a: <reason>")

system, not the code. Re-derive assumptions from scratch; question the
architecture; consider that the bug is somewhere you have not looked.
